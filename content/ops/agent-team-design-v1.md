这份文件是当前团队型 agent 系统的第一版正式设计。目标不是只描述“有哪些角色”，而是定义一套真正可执行、可扩展、可验收的协作机制。

## 一、设计目标

1. 让重要任务不再依赖单线程临时记忆。
2. 让不同类型的任务由不同特长角色处理。
3. 让所有结果都有任务归属、状态、责任人与最终归档位置。
4. 让 Leader Agent 可以统一调度、验收、汇报，而不是亲自承担所有细碎工作。

## 二、设计原则

### 1. 先登记，后执行
新任务必须先进入任务总表，再开始执行。

### 2. 先拆分，后分派
复杂任务必须先拆成子任务，再交给对应专长角色。

### 3. 先验收，后汇报
子任务完成后先回 Leader Agent，不直接视为已完成。

### 4. 先落盘，后遗忘
重要决策、长期规则、正式结论都必须写入文件，不依赖聊天上下文。

## 三、团队角色构成

### A. Leader Agent
定位：团队负责人 / 调度中心 / 最终验收人

核心职责：
- 接收老大指令
- 判断任务类型与优先级
- 创建任务卡
- 拆解为子任务
- 指派给对应角色
- 汇总结果并形成最终回复
- 决定记忆沉淀与 GitHub 发布动作

适合处理：
- 跨领域复杂任务
- 多阶段交付任务
- 需要取舍和优先级判断的任务

### B. Research Agent
定位：研究员 / 方案分析员

核心职责：
- 搜集公开资料与社区讨论
- 提取核心观点与反例
- 对比多个方案优劣
- 起草研究文件主体

适合处理：
- 社区趋势研究
- 技术方案对比
- 内容调研
- 市场与产品分析

### C. Debug Agent
定位：排障工程师 / 根因分析员

核心职责：
- 复现问题
- 阅读日志与状态输出
- 分析异常路径
- 提出最小修复与缓解方案

适合处理：
- server_error 排查
- memory 异常
- gateway / config / runtime 故障

### D. Memory Agent
定位：任务记录员 / 连续性管理员

核心职责：
- 维护任务总表
- 更新 MEMORY.md 与 daily memory
- 记录长期规则
- 检查是否有任务遗漏或状态过期

适合处理：
- 重要规则落盘
- 任务连续性维护
- 任务卡补全
- 历史回溯

### E. Publishing Agent
定位：发布编辑 / 仓库归档员

核心职责：
- 把草稿整理为正式仓库文件
- 对齐内容格式与 front matter
- 检查链接、来源、摘要是否齐全
- 执行 commit / publish 前整理

适合处理：
- 博客文章
- 研究报告
- 排查报告
- 规范文档

## 四、任务类型与分派矩阵

### 1. 研究类任务
默认负责人：Research Agent
协同角色：Publishing Agent / Leader Agent

标准产物：
- 研究文件
- 方案对比
- 结论与论据
- 来源与链接列表

### 2. 排障类任务
默认负责人：Debug Agent
协同角色：Research Agent / Leader Agent

标准产物：
- 复现路径
- 根因分析
- 修复建议
- 风险与缓解方案

### 3. 规则/记忆类任务
默认负责人：Memory Agent
协同角色：Leader Agent

标准产物：
- 任务卡
- 任务板更新
- 长期规则更新
- 记忆归档记录

### 4. 发布类任务
默认负责人：Publishing Agent
协同角色：Research Agent / Leader Agent

标准产物：
- 仓库正式文件
- 已提交版本
- 发布记录

## 五、标准状态流转

每个任务都必须经过以下状态之一：
- TODO：已登记，未开始
- DOING：进行中
- BLOCKED：阻塞中
- REVIEW：等待 Leader 验收
- DONE：已完成并归档

推荐流转：
TODO → DOING → REVIEW → DONE
如遇问题：
DOING → BLOCKED → DOING

## 六、标准执行流程

### 第一步：任务进入系统
Leader Agent 接到任务后，先做三件事：
1. 命名任务
2. 判断任务类型
3. 登记到任务总表

### 第二步：创建任务卡
任务卡至少包含：
- 任务编号
- 任务名称
- 类型
- 目标
- 交付标准
- 负责人
- 当前状态
- 下一步动作
- 最终归档位置

### 第三步：角色分派
根据任务类型，把任务交给对应专长角色主责推进。

### 第四步：结果回收
所有中间结果统一回到 Leader Agent，由 Leader 判断：
- 是否补证据
- 是否需要二次排查
- 是否达成完成标准

### 第五步：归档与发布
正式完成的任务，至少进入两个地方：
1. 任务板状态更新
2. 仓库正式文件或记忆文件更新

## 七、研究任务强制格式

凡是研究任务，正式文件必须包含：
1. 结论
2. 论据
3. 文件来源
4. 链接
5. 摘要

缺任何一项，都不算正式完成。

## 八、当前落地方式

当前环境下，团队 agent 先采用“两阶段落地”：

### 阶段 1：角色化执行
当运行环境不适合长期挂多个独立子 agent 时，由 Leader Agent 按角色拆分并串行推进，但在文档与任务板中保留明确的角色归属。

### 阶段 2：子 agent 化执行
当任务规模进一步增加，或需要并发处理多个长期任务时，再把这些角色升级为独立 session / 子 agent：
- Research 子 agent
- Debug 子 agent
- Memory 子 agent
- Publishing 子 agent

届时 Leader Agent 只负责：
- 发任务
- 收结果
- 做验收
- 对老大汇报

## 九、当前明确的构建任务

为了让这套系统真正运转，当前还需要持续推进以下构建项：
1. 完善任务板
2. 建立研究模板
3. 建立任务卡模板
4. 确定子 agent 启动/协作机制
5. 形成定期汇报格式

## 十、v1 结论

这套设计的核心不是“多造几个 agent 名字”，而是建立一个可执行的任务操作系统：
- Leader 负责统筹
- 专长角色负责推进
- 任务板负责跟踪
- 研究文件负责沉淀
- GitHub 负责长期归档

这是当前最适合本工作流的第一版团队型 agent 设计。
